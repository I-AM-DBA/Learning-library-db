# I AM DBA

서울도서관 홈페이지의 UI/UX로부터 Database의 스키마를 추측하고 간략히 구현해보는 팀 스터디입니다.

## 구성원

- 박종현 (팀장)
- 남보람
- 심주은
- 옥재호
- 조승현

## 목표 ☆

> DB 테이블의 목적 및 성능을 고려한 테이블 설계 이해하기

## 주차별 학습 계획

- 1주차
  - 기초 DB 설계 이론을 학습
  - 요구사항 분석 및 기능 정의서 작성
  - 서울시도서관 홈페이지 UI/UX를 바탕으로 ERD 초안 도출
- 2주차
  - DDL SQL, 제약조건, DBMS 학습을 통해 제2정규화 구현
  - 도출한 ERD 초안을 이용하여 논리적/물리적 설계 및 DBMS 서버 구축
  - DBMS에 도서관 공공데이터 입력 및 성능 테스트 진행
- 3주차
  - 제3정규화를 학습하고 이에 맞추어 1차 스키마 구조 최적화를 진행
  - 개선한 테이블에 맞춰 공공데이터를 입력하고 성능 테스트를 진행
- 4주차
  - 피드백을 반영하여 2차 스키마 구조 최적화 진행
  - 최종 스키마를 이용하여 성능 테스트 진행
  - Optional
    - 조회 SQL 최적화 시도

## 결과물 (가이드라인) ☆

- ERD (Entity Relationship Diagram)
- 단계별 DB 테이블 스키마
- 주차별 성능 테스트 보고서
  - 2주차: 초안 ERD의 스키마와 제2 정규화를 적용한 테이블의 성능 보고서
  - 3주차: 제3 정규화 이론을 반영한 스키마의 성능 보고서
  - 4주차: 피드백을 반영한 스키마의 최종 성능 보고서
- Optional
  - 조회 SQL 쿼리 최적화 시도 및 결과 보고서

## OKR (Objectives and Key Results)

1. 최종 스키마 기반 기능 정의서에 대한 상호이해도 확인(1주차 기능 정의서 작성 기반)
1. 성능 테스트 결과 및 피드백 반영 기반 향상도 체크 -> 최종 리포트 체크
1. Optional
   - 조회 SQL

## Git 브랜치 전략 ☆

- 대상 전략: GitHub Flow
- 브랜치 전략 선정 이유
  1. 프로젝트 규모가 작고 불필요하게 많은 브랜치가 필요하지 않을 것으로 예상함
  1. 복잡한 브랜치 전략의 경우 전략 자체에 대한 이해와 실천에 불필요한 노력이 필요할 것이 우려됨
  1. 쉽고 빠르게 학습할 수 있는 전략이 선호됨
- 브랜치 네이밍 규칙: 타입 / 기능대상

    - `feat/*`
