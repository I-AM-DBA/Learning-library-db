# I AM DBA

서울도서관 홈페이지의 UI/UX로부터 Database의 스키마를 추측하고 간략히 구현해보는 팀 스터디입니다.

## 구성원

- 박종현 (팀장)
- 남보람
- 심주은
- 옥재호
- 조승현

## 목표 ☆

> DB 테이블의 목적 및 성능을 고려한 테이블 설계 이해하기

## 주차별 학습 계획

- 1주차
  - 기초 DB 설계 이론을 학습
  - 요구사항 분석 및 기능 정의서 작성
  - 서울시도서관 홈페이지 UI/UX를 바탕으로 ERD 초안을 도출
- 2주차
  - DDL SQL, 제약조건, DBMS 학습을 통해 제2정규화 구현
  - 도출한 ERD 초안을 이용하여 논리적/물리적 설계 및 DBMS 서버 구축
  - 도서관 오픈API 데이터를 기반으로 DBMS에 입력하고 성능 테스트 진행
- 3주차
  - 정규화와 반정규화, 인덱스를 학습 및 1차 스키마 구조 최적화를 진행
  - 개선한 테이블에 다시 공개 데이터를 입력하고 성능 테스트를 진행
- 4주차
  - 피드백을 반영하여 2차 스키마 구조 최적화 진행
  - 최종 스키마를 이용하여 성능 테스트 진행
  - Optional
    - 조회 SQL 최적화 시도

## 결과물 (가이드라인) ☆

- ERD (Entity Relationship Diagram)
- 단계별 DB 테이블 스키마
- 주차별 성능 테스트 보고서
  - 2주차: 초안 ERD를 이용하여 구성한 스키마와 성능 보고서
  - 3주차: 정규화 이론을 반영한 스키마의 성능 보고서
  - 4주차: 피드백을 반영한 스키마의 최종 성능 보고서
- Optional
  - 조회 SQL 쿼리 최적화 시도 및 결과 보고서

## OKR (Objectives and Key Results)

1. 최종 스키마 기반 기능 정의서에 대한 상호이해도 확인(1주차 기능 정의서 작성 기반)
1. 성능 테스트 결과 및 피드백 반영 기반 향상도 체크 -> 최종 리포트 체크
1. Optional
   - 조회 SQL

## Git 브랜치 전략 ☆

- 브랜치 전략 선정 이유
  - 대상 전략: GitHub Flow
  - 구성:  `main`, `develop`, `feature`, `fix` 브랜치만으로 구성함.
  - 사유:
        1. 프로젝트 특성 고려
        2. 깃 사용에 대한 이해  ~~~ 바탕으로 팀원간 협업 ~~~~ 이슈 최소화 및 숙련도 향상
        - 모두가 처음 깃을 사용
        - 모두 깃을 이용한 협업 경험이 처음
        - `develop` 브랜치를 추가한 사유 명시 필요
        => 이슈 최소화하고 깃을 이용한 협업의 목적성을 이해하고 싶음.
- 브랜치 네이밍 규칙: 타입 / 기능대상
  - `main`
    - `develop`
      - `feat/search`
      - `feat/reserve`
      - `fix/search`
      - `fix/reserve`
